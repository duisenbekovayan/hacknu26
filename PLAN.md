# План разработки: цифровой двойник локомотива (Go + PostgreSQL + WebSocket)

## Идея старта: «искусственные датчики»

Да, это хороший порядок работ: сначала **стабильный генератор** (константы или простой паттерн), который непрерывно шлёт данные на бэкенд. Так вы проверяете приём, схему БД, рассылку по WebSocket и UI без шума от сложной математики. На втором шаге генератор заменяется или дополняется **реалистичным сценарием** (разгон, инцидент, восстановление) и режимом highload.

## Фазы

### Фаза 0 — каркас (текущий спринт) ✅

- Docker Compose: PostgreSQL (`docker-compose.yml`).
- Бэкенд Go: `POST /api/v1/telemetry` (ingest), `GET /latest`, `GET /history`, `GET /ws/telemetry`, `GET /healthz`; схема `internal/db/schema.sql`, JSONB + индекс здоровья.
- Симулятор «датчиков»: `simulators/cmd/simulator` + пакет `simulators/synth` — генерация в памяти, 1 Гц.
- OpenAPI: `openapi.yaml`, запуск: `README.md`.

### Фаза 1 — продуктовые требования кейса

- Конфиг порогов и весов в YAML/ENV, пересчёт индекса без перекомпиляции.
- REST: история за окно (replay 5–15 мин), экспорт CSV.
- Клиент: дашборд «Кабина», графики, темы, индикатор «нет связи» (reconnect/backoff).

### Фаза 2 — нагрузка и надёжность

- Имитация x10 событий (несколько поездов или ускоренный симулятор).
- Очередь/буфер (упрощённо: канал + батчи в БД).  
  Стартовая версия уже внедрена: bounded in-memory queue + worker pool для ingest.
- Базовая аутентификация для админ-эндпоинтов.

### Фаза 3 — демо

- Docker Compose «одной командой»: Postgres + API + (опционально) фронт.
- Скрипт нагрузочного сценария, запись короткого видео.
